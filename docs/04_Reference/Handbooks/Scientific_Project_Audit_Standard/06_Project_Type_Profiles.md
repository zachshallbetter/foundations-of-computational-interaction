# Project-type audit profiles

## General rule

Project type determines criterion applicability and weights, not the definition of scientific validity. Auditors must not penalize a project for omitting evidence it does not claim to provide. They must also not credit a project for a stronger scientific state than its evidence establishes.

## Conceptual paper

Emphasize:

- problem significance;
- contribution clarity;
- definitions;
- ontology;
- semantic precision;
- counterexamples;
- comparison with alternatives;
- discriminating consequences;
- scope discipline.

Required gates normally include object coherence, claim-evidence compatibility, reproducibility of argument, and integrity. Measurement and empirical-design gates apply when the paper makes measurement or empirical claims.

## Theory

Emphasize:

- explanatory target;
- primitives;
- assumptions;
- mechanism or law structure;
- derivations;
- scope;
- risky consequences;
- compatibility with established evidence;
- comparison with rival theories.

A theory must not receive empirical credit solely for accommodating known observations after the fact.

## Formal model

Emphasize:

- formal syntax and semantics;
- defined symbols;
- consistency;
- completeness relative to scope;
- identifiability;
- tractability;
- invariants;
- edge cases;
- correspondence with observables;
- sensitivity and parameter dependence.

## Ontology or taxonomy

Emphasize:

- category identity;
- inclusion and exclusion rules;
- exhaustiveness relative to scope;
- overlap and hierarchy;
- relation types;
- counterexamples;
- inter-rater application;
- incremental value over existing schemes.

Conceptual distinctions should be tested for reliable application and scientific usefulness.

## Measurement system or instrument

Emphasize:

- construct definition;
- measurand;
- observation model;
- scale;
- reliability;
- validity;
- invariance;
- calibration;
- uncertainty;
- missingness;
- interpretation limits.

A reliable instrument that measures the wrong construct fails validation.

## Empirical study

Emphasize:

- sampling;
- operationalization;
- design;
- identification;
- comparison conditions;
- evidence provenance;
- estimand-estimator alignment;
- uncertainty;
- robustness;
- preregistered deviations;
- generalization limits.

## Computational implementation

Emphasize:

- correspondence between theory and code;
- deterministic and inspectable transformations;
- validation data;
- leakage prevention;
- failure behavior;
- reproducibility;
- benchmark choice;
- ablation;
- robustness;
- resource and maintenance burden.

Passing software tests establish implementation conformance only to the tested specification.

## Dataset

Emphasize:

- sampling frame;
- provenance;
- consent and governance;
- inclusion and exclusion;
- coding;
- missingness;
- representativeness;
- leakage;
- versioning;
- documentation;
- intended and prohibited uses.

## Benchmark

Emphasize:

- construct validity;
- task coverage;
- contamination;
- baseline selection;
- scoring rule;
- ceiling and floor effects;
- uncertainty;
- robustness to implementation details;
- relationship between benchmark performance and claimed capability.

## Replication

Emphasize:

- correspondence to the target study;
- justified deviations;
- protocol reproducibility;
- analytic reproducibility;
- population and setting differences;
- equivalence or smallest-effect bounds;
- heterogeneity;
- failure classification;
- transport claims.

## Literature review or synthesis

Emphasize:

- review question;
- search strategy;
- inclusion and exclusion;
- source quality;
- coding;
- evidence weighting;
- publication bias;
- conceptual harmonization;
- heterogeneity;
- traceability from synthesis to sources.

## Research program

Emphasize:

- dependency structure;
- coherence across projects;
- canonical terminology;
- cumulative evidence coverage;
- unresolved foundational risks;
- duplication;
- prioritization;
- comparability;
- maintenance burden;
- theory-to-evidence progression.
