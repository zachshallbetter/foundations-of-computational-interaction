# Foundations of Computational Interaction Editorial Style Guide

## Authority and scope

The Knowledge System of Record governs terminology and definitions. Research papers and infrastructure artifacts provide provenance, evidence, and implementation guidance. Editorial changes may improve presentation but may not alter ontology, first principles, contracts, or formal semantics.

## Language

- Use American English.
- Use the Oxford comma.
- Prefer direct, declarative sentences.
- Use ASCII hyphens and dashes in machine-readable and cross-platform artifacts.
- Avoid contractions in normative and technical prose.

## Canonical terms

- Preserve the exact spelling of canonical concepts.
- Capitalize named artifacts: Projection Contract, Opportunity Contract, Interaction Contract, Behavioral Evidence Contract, Measurement Contract, Similarity Contract, Transfer Contract, Strategy Contract, Prediction Contract, and Comparison Contract.
- Use lower case for generic mentions of concepts unless beginning a sentence or naming an artifact.
- Do not create local synonyms or redefine canonical concepts.

## Headings

- Use one level-1 heading per standalone document.
- Use sentence case for section headings.
- Do not skip heading levels.
- Separate headings from body text with one blank line.

## Lists and tables

- Use bullets for parallel items and numbered lists for procedures or ranked sequences.
- Keep list items grammatically parallel.
- Use sentence punctuation for complete sentences; omit terminal punctuation for short fragments.
- Give every table a descriptive title when presented in publication form.

## Figures

- Assign stable identifiers in the form `FIG-NNN`.
- Use captions in the form `Figure N. Descriptive sentence.`
- Record source artifact, editable source, export formats, version, concept links, rights status, and accessibility text.
- Refer to figures by number, not by position.

## Equations, notation, and code

- Define every symbol at first use and preserve the canonical Symbol Table.
- Number equations only when referenced later.
- Use fenced code blocks with a language identifier where applicable.
- Preserve literal identifiers, file paths, citation keys, and schema fields exactly.

## Citations and references

- Use the Master Bibliography as the system of record.
- Use one canonical citation key per work.
- Do not infer missing bibliographic metadata.
- Prefer primary and authoritative sources.

## Change control

- Classify changes as Editorial, Structural, or Normative.
- Normative changes are prohibited in Version 1.1.1.
- Record theoretical improvements only in Appendix A - Candidate Version 2.0 Changes.
